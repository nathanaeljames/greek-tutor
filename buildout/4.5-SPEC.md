# 4.5-SPEC.md — Phase 4.5: Audio storage migration to IndexedDB (Claude Code work order)

Prerequisites: CHAT-HANDOFF.md (the round 1-3 storage saga and the
DECISION section), HANDOFF-4.md sections 8-9.5. Current code state:
commit ba5d0d4. Commit any pending work first. Output: HANDOFF-4.5.md
in buildout/.

## 0. Decision record and the evidence it rests on

DECIDED (Fable, 2026-07-19): audio bytes move from Cache Storage to
IndexedDB before phase 5. Basis: the round-3 cold-start measurement —
resp-start ~4000ms with js-start immediately after, i.e. the entire
delay is inside the SW fetch handler while WebKit brings up the
origin's Cache Storage backing store (ONE database per origin, not per
named cache) holding ~8521 audio entries. No app-code change can
remove a cost that is paid before app code runs; the only lever is
making the origin's Cache Storage small again. Rejected alternative:
capping Download All to built chapters only — that merely defers the
same floor until phase 5 builds the chapters out (~8.5k files either
way).

Target state: Cache Storage holds ONLY the app shell precache (~15
files) + the manifest cache (1 entry) — fast to bring up forever,
regardless of audio library size. Audio bytes live in IndexedDB as
Blobs. Cold start must not scale with library size (this generalizes
standing directive 10).

## 1. The Range question is dissolved, not reimplemented

The CHAT-HANDOFF flagged Range-request serving as the known unknown.
Scoping answer: playback moves to Blob object URLs, and Range ceases
to exist as a concern. Range is an HTTP-layer concept; iOS issues
ranged requests only because playback currently goes over HTTP URLs
through the SW. An <audio> element whose src is
URL.createObjectURL(blob) seeks natively against local bytes with no
fetch, no SW, and no Range header anywhere. Consequences:

  - The SW gets ENTIRELY out of the audio path: remove the /audio/
    CacheFirst runtime route AND the workbox rangeRequests plugin
    usage from vite.config.js. Keep the app-shell precache and the
    manifest NetworkFirst route exactly as they are.
  - The x-gt-bulk-download marker header and the SW-exclusion logic
    from round 1 become obsolete — remove them. Single-writer is now
    structural (nothing but the app writes audio bytes anywhere).
  - The force-update delete-then-fetch dance (which existed to bypass
    the SW's CacheFirst copy) simplifies to: delete IDB keys, fetch,
    store.
  - Clip sizes make whole-blob handling trivial (~10-60 KB typical,
    the 27s alphabet recitation ~100 KB). Nothing here needs
    streaming.

What is genuinely lost and accepted: SW-transparent audio (any
<audio src="/audio/..."> tag would previously have worked anywhere).
All playback must go through audio.js — which is ALREADY the sole
audio choke point (verified since HANDOFF-4B §5), so nothing actually
breaks. Assert this in the handoff after a grep for any stray /audio/
URL usage outside audio.js/downloads.js.

## 2. New module: src/lib/audio-store.js

The single IndexedDB access point. Nothing else in the codebase may
open the database.

  - Database 'greek-tutor', object store 'audio', keys = the same
    absolute paths used today ('/audio/chapt_1/a_alpha.m4a') so
    migration and all existing id->path plumbing are symmetric.
    Values: Blobs (store the Blob itself, not ArrayBuffer — no copy
    into JS memory; iOS IDB Blob support is solid on the target OS).
  - API (all promise-based): getBlob(path), putMany(entries) (batched
    transactions, ~100 per txn), deleteMany(paths), clearAll(),
    count(), keys(prefix?). Lazy DB open on first call.
  - Library: use `idb` (npm install idb). What it is: Jake
    Archibald's ~1.2 KB promise wrapper over raw IndexedDB, no
    further dependencies. Why: raw IDB's event/transaction callback
    model is a well-known footgun mine (auto-committing transactions,
    error propagation); idb removes that class of bug for the cost of
    a kilobyte. It is the de-facto standard wrapper and phase 6's
    progress backend will reuse it. If there is a strong reason to
    hand-roll instead, note it in the handoff — but batched
    multi-put transactions are exactly where hand-rolled IDB code
    goes wrong.

## 3. downloads.js rework

  - bulkFetch() keeps its 25s timeout + 2 retries + 429/503 backoff
    (round-2 fix, load-bearing). Fetched blobs go to
    audioStore.putMany in batches; drop the upfront cache.keys()
    snapshot in favor of one audioStore.keys() read per run
    (skip-if-present resume semantics unchanged).
  - Pack state/badges: unchanged pattern — localStorage bookkeeping +
    the persisted audioCount store render instantly; NOTHING on the
    load or route-mount path may enumerate the store (directive 10
    stands; idb count() is cheap but the discipline stays —
    reconciliation remains Settings-only via reconcileAudioCache(),
    now backed by count()/keys() and effectively instant).
  - clearAllAudio(): audioStore.clearAll() + bookkeeping reset + ALSO
    delete the legacy 'greek-tutor-audio' cache if it still exists
    (belt-and-braces during the migration window).
  - Update flow: 'update' state logic unchanged; force path =
    deleteMany(pack paths) then normal download.
  - Cancel/resume semantics unchanged from the user's perspective.

## 4. audio.js rework

play(id) resolves the path, then:
  - IDB hit: createObjectURL, play. Revoke the URL on ended, on
    stop(), and before creating the next one (no URL leaks; keep at
    most one live object URL).
  - IDB miss + online: plain fetch(path) (reuse the timeout helper),
    audioStore.putMany([entry]), then play — preserving today's
    first-play-while-online-caches-for-offline parity exactly.
  - IDB miss + fetch fails/offline: toast. The round-1 contract is
    LAW: toast if and only if the user gets no audio.
Stop-on-route-change behavior unchanged.

## 5. One-time migration of existing installs

Fable's iPhone and iPad hold the full ~8.5k-entry library (plus
round-1-era Vary duplicates) in the legacy cache. Requirements:

  - Runs deferred after first paint (idle-scheduled from the existing
    deferred-init hook) — NEVER on the load/mount path. Chunked
    (~100 entries: read response -> blob -> putMany -> delete those
    cache entries), yielding between chunks. Idempotent and
    resumable: keys already in IDB are skipped, an interrupted run
    picks up the remaining cache entries next launch.
  - Vary duplicates dedupe naturally (same IDB key overwrites); the
    migrated count is the DISTINCT path count, which should finally
    make the file count exact and stable on iOS.
  - On completion: caches.delete('greek-tutor-audio'), set a done
    flag in the downloads bookkeeping, refresh audioCount.
  - Settings shows a one-line migration status while it runs
    ("Optimizing audio storage... n of m"); the diagnostic card gains
    IDB stats (store count, done flag) and KEEPS the cold-start
    metric — it is the acceptance instrument.
  - Known and accepted: the FIRST cold start on the upgrade build
    still pays the old Cache Storage bring-up once (it happens before
    JS; nothing we can do), and the migration itself pays the legacy
    scan cost once in the background. After the legacy cache is
    deleted, cold start is permanently fast. State this in the
    handoff so the first post-deploy launch isn't misread as a
    regression.
